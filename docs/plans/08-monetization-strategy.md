# 08. 수익화 전략 (Monetization Strategy)

상태: 백엔드 일부 완료 — 서버비 프로그레스 바 API. 광고/포인트 시스템은 대기 (아래 "백엔드 구현 현황" 참고)

## 백엔드 구현 현황 (2026-07-16)

| 항목 | 상태 |
|---|---|
| Phase 1~2 네이티브 광고 | 백엔드 작업 없음 — 광고 삽입 위치/스켈레톤 UI는 전부 프론트 영역. 애드센스/애드몹 연동도 프론트+퍼블리셔 계정 설정이라 백엔드 대상 아님 |
| Phase 3~ 포인트 시스템 + 보상형 비디오 | **미착수** — 포인트 원장(적립/차감 이력), 뱃지/프레임/이모티콘 보유 및 상점, 보상형 광고 SDK 연동 등 구체적인 데이터 모델이 이 기획서에 아직 정의돼 있지 않음. 착수하려면 먼저 포인트 획득/차감 이벤트 종류, 상점 아이템 목록, 광고 SDK(AdMob 등) 선정부터 기획 확정이 필요 — 지금 임의로 스키마를 설계하면 이후 기획 변경 시 갈아엎어야 할 위험이 커서 보류
| 서버비 충당 프로그레스 바 | **완료** — `ServerCostSettingEntity`(id=1 고정 싱글톤 행: targetAmount/currentAmount). `GET /api/settings/server-cost`(공개, 게이지 UI용 — targetAmount/currentAmount/percentage 0~100 반환), `GET/PUT /api/admin/settings/server-cost`(관리자, 06번 문서에서 예고된 API 이름 그대로). percentage는 목표 초과 달성 시 100에서 클램프(원본 금액은 그대로 노출) |

### 프론트 연동 시 필요한 것

- 메인/사이드바 게이지: `GET /api/settings/server-cost` 호출 → `percentage`로 진행 바, `currentAmount`/`targetAmount`는 "780,000 / 1,000,000원" 같은 보조 텍스트에 사용
- 관리자 설정 화면: `GET /api/admin/settings/server-cost`로 현재 값 프리필 → 폼 제출 시 `PUT`으로 갱신(기존 `GameManagementPage` 등과 동일한 단일 리소스 폼 패턴)
선행 조건: 플랫폼 기본 기능 오픈 (Phase 1~2 이후 점진적 적용)

## 배경 및 목적

Gacha Scheduler 플랫폼의 유지보수 및 서버 운영 비용을 충당하고, 장기적인 서비스 생존을 도모하기 위해 수익화 모델을 도입합니다. 단, 모바일 앱(하이브리드 앱) 환경에서의 UX 훼손을 최소화하고 커뮤니티 성질에 맞는 거부감 없는 수익화 기조를 유지합니다.

## Phase 별 수익화 기획

### [Phase 1~2] 초기 수익화: 네이티브 광고 (In-Feed Ads)

초기 런칭 시점에는 유저 모객과 이탈률 방어가 최우선이므로, UX를 방해하는 강제적인 전면 광고나 모바일 화면을 가리는 상/하단 고정 배너(Sticky Banner)는 사용하지 않습니다.

*   **방식**: 구글 애드센스(Google AdSense) 또는 구글 애드몹(AdMob) 기반의 **네이티브 광고(Native Ads)**
*   **노출 위치**:
    *   **공략 게시판 목록**: 게시글 7~10개당 1개 꼴로 게시글과 완전히 동일한 UI 디자인 형태의 광고 삽입.
    *   **일정 스케줄러 목록**: 리스트 뷰에서 일정 항목 사이에 자연스럽게 배치.
    *   **PC 웹 한정**: 우측 사이드바 하단 플로팅 배너 및 게시글 본문 최하단 배너 허용 (PC는 여백이 많아 UX 저하가 적음).
*   **제약 사항**: 모바일 기기(하이브리드 앱 포함)에서는 하단 탭바(Bottom Navigation)의 터치 오작동을 막기 위해 하단 고정 배너를 절대 사용하지 않습니다.

### [Phase 3~] 고도화 수익화: 포인트 시스템 + 보상형 비디오 (Rewarded Video)

커뮤니티 트래픽이 안정화되고 유저의 소속감이 생기는 시점(Phase 3)부터 내부 재화(포인트) 시스템을 도입하여 보상형 광고와 연동합니다. 강제 기능 제한이 아닌 '꾸미기' 위주의 보상을 제공합니다.

*   **포인트(재화) 획득처**:
    *   커뮤니티 활동 (글 작성, 베스트글 선정, 출석체크 등)
    *   **보상형 비디오 시청** (유저가 원할 때 광고 1회 시청 시 50 포인트 지급 등)
*   **포인트(재화) 사용처 (꾸미기 보상)**:
    *   **프로필 커스텀**: 닉네임 옆 전용 뱃지, 프로필 테두리 프레임 장착
    *   **커뮤니티 이모티콘**: 공략 게시판 댓글에 사용할 수 있는 움직이는 이모티콘 구매
    *   **가챠 공유 카드 스킨**: 07 파트에서 기획된 가챠 결과 이미지 생성기에서 '프리미엄 스킨' 적용
*   **효과**: 광고 단가가 가장 높은 보상형 비디오(Rewarded Video)를 유저의 자발적 선택으로 시청하게 유도하여 UX 저하 없이 수익률을 극대화합니다.

### 커뮤니티 결속력 강화: 서버비 충당 프로그레스 바 (Progress Bar)

수익 창출에 대한 유저들의 거부감을 줄이고, '우리가 함께 커뮤니티를 운영한다'는 소속감과 투명성을 제공하기 위해 시각적인 목표 달성 게이지를 추가합니다.

*   **방식**: 관리자 수동 제어 (애드센스/후원 등 다채널 수익을 고려하여 자동 API 연동 대신 수동 입력 채택)
*   **관리자 페이지(백엔드/프론트)**: '이번 달 서버 유지비 목표액' 및 '현재 달성액(또는 %)'을 입력/수정하는 기능 제공.
*   **유저 화면(프론트)**: 메인 스케줄러 상단이나 사이드바 영역에 "이번 달 서버비 🟢 78% 달성!"과 같은 귀여운 게이지 UI 노출.
*   **효과**: 애드블록(AdBlock) 자발적 해제 유도 및 커뮤니티 소속감 고취.

## 개발 참고 사항

- 프론트엔드 작업 시, Phase 1의 네이티브 광고가 들어갈 수 있는 빈 영역(Skeleton UI 또는 Placeholder)을 리스트 컴포넌트 중간에 삽입할 수 있도록 확장성을 고려하여 컴포넌트를 설계해야 합니다.
- 앱 셸(Capacitor) 환경과 일반 모바일 웹 브라우저 환경에서 AdSense 스크립트 충돌이 없는지 배포 전 사전 테스트가 필요합니다.
- 서버비 프로그레스 바 데이터는 관리자 설정 테이블(가칭 `SiteSettings` 등)에 저장하여 전역 상태로 불러오도록 설계합니다.
